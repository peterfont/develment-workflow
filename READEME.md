## 这是瀑布流吗？和 AI 时代的关系

### 是的，骨子里有瀑布流的影子

上面的流程确实是线性的：设计 → 开发 → 上线。这被称为**瀑布模型（Waterfall）**，是1970年代提出的，最大的问题是：

```
阶段之间没有反馈
需求没想对 → 开发完了才发现 → 返工成本极高
```

### 现代范式是怎么做的

**1. 敏捷开发（Agile / Scrum）**
```
不是做完所有再上线
而是每 1-2 周一个迭代（Sprint）
每个迭代都交付一个可用的小功能
持续收集用户反馈 → 修正方向

适合：有团队、有固定用户、需求经常变
```

**2. Shape Up（Basecamp 提出）**
```
6周为一个周期
前2周：老板/产品定义"问题范围"（不是具体方案）
后4周：开发者自己决定怎么做
不允许无限延期，到期就停

特点：给开发者更多自主权，避免需求蔓延
适合：小团队，产品方向相对稳定
```

**3. 精益创业（Lean Startup）**
```
Build（快速构建最小版本）
  → Measure（测量用户反应）
    → Learn（学到了什么）
      → 重复

核心：不要在没有验证的功能上浪费时间
适合：0到1阶段，还不确定用户要什么
```

**4. Vibe Coding（AI 时代新范式）**
```
不写设计文档，直接用 AI 生成原型
快速可视化 → 给用户看 → 收集反馈 → 再迭代
代码质量先不管，先验证方向

优点：极快，适合验证想法
缺点：技术债积累极快，扩展困难
适合：个人项目、Demo、MVP 快速验证
```

### AI 时代改变了什么，没改变什么

| | 过去 | AI 时代 |
|--|------|---------|
| 写一个注册接口 | 2小时 | 10分钟 |
| 写前端页面 | 半天 | 1小时 |
| 查文档、查语法 | 慢 | 几乎消失 |
| **想清楚做什么** | 要花时间 | **还是要花时间** |
| **数据库设计错了** | 要重改 | **还是要重改** |

**AI 加速的是执行，不是决策。**

```
以前：想错了 → 写3天烂代码 → 返工3天
现在：想错了 → AI帮你写3小时烂代码 → 返工还是要3天
```

---

### 对小团队/个人开发者的建议：混合范式

本文档的流程不是严格瀑布流，而是**轻量版迭代**：

```
Phase 0-2（设计）：  花 1-2 天想清楚，不追求完美
                    ↓
Phase 3-4（后端核心）：先跑通主链路
                    ↓
上线最小版本，收集反馈
                    ↓
Phase 5+（迭代）：   根据真实用户反馈再加功能
```

关键原则：
- **不要在没有用户的情况下过度设计**（Lean 思想）
- **数据库表要设计稳，其他都可以随时改**（务实）
- **每个阶段结束都要有可交付物**（Agile 思想）
- **用 AI 加速执行，把省下的时间用在决策上**（AI 时代红利）

## Phase 0 — 需求设计

**目标：把"想法"变成可执行的描述**

### 做什么
- 列出所有角色（用户是谁）
- 用"用户故事"描述每个功能：
  > "作为 **用户**，我想要 **输入激活码续费**，以便 **延长使用时间**"
- 画出核心流程图（哪怕是草稿）
- 确认边界：哪些做，哪些不做

### 不做什么
- 不画 UI 图（太早，需求还会变）
- 不写代码

### 完成标志
- 能用一句话说清楚每个功能
- 知道有几种用户角色，每个角色能做什么

### 常见错误
- ❌ 跳过这步直接开始写代码 → 写到一半发现方向错了
- ❌ 需求太细太完美 → 浪费时间，先做核心流程

## Phase 1 — 数据设计（最重要）

**目标：设计数据库表结构**

### 为什么最重要
```
表是地基
表改了 → 接口要改 → 前端要改 → 全部返工
表设计稳了 → 后面怎么改代码都不影响数据
```

### 做什么
1. 列出所有"实体"（名词）
   - 用户、卡密、产品、激活记录、渠道商...
2. 确定实体之间的关系
   - 一个用户可以有多张激活记录（一对多）
   - 一张卡密只能被一个用户激活（一对一）
3. 设计每张表的字段
   - 每个字段叫什么名字
   - 是什么类型（字符串/数字/时间）
   - 是否可以为空

### 表设计原则
```
1. 字段命名统一（snake_case：created_at 不是 createdAt）
2. 每张表都要有 id、created_at
3. 能用外键关联就用外键，不要把所有东西塞一张表
4. 不确定的字段用 JSON 类型存（方便扩展）
5. 状态类字段用 ENUM（'active','disabled'），不要用数字
```

### 完成标志
- 画出 ER 图（实体关系图）或写出完整建表 SQL
- 每张表能回答：**这张表存什么，和哪张表有关系**

## Phase 2 — 接口设计

**目标：定义前后端之间的"合同"**

### 做什么
不用写代码，只需要列出来：

```
接口名称   方法    路径                  说明
─────────────────────────────────────────────
注册       POST    /api/auth/register    邮箱+密码注册
登录       POST    /api/auth/login       返回 JWT token
激活       POST    /api/activate         用激活码换有效期
鉴权检查   GET     /api/auth/check       插件启动时调用
```

每个接口还需要定义：
- **请求参数**：发什么数据过去
- **响应格式**：返回什么数据
- **鉴权要求**：需要登录吗

### 完成标志
- 前端开发者看到这份清单，知道调什么接口、传什么参数、拿到什么数据
- 后端开发者看到这份清单，知道要实现什么

### 重要原则
```
接口设计是前后端之间的合同
双方确认后，可以并行开发（前端 mock 数据，后端写实现）
```

## Phase 3 — 后端实现

**目标：让接口真正能跑通**

### 做什么
按接口清单逐个实现：
1. 连接数据库
2. 写路由和业务逻辑
3. 处理错误（参数缺失、数据不存在等）
4. 写完一个，用 Postman 测一个

### 开发顺序
```
先写"主链路"上的接口
  → 注册 → 登录 → 激活 → 鉴权
后写"辅助"接口
  → 修改密码、查询记录、管理后台接口...
```

### 完成标志
- 核心接口全部 Postman 测通
- 错误情况有合理的返回（不是直接 500）

### 常见错误
- ❌ 边写边改数据库表结构 → 重新执行 Phase 1
- ❌ 没有错误处理，接口直接崩溃
- ❌ 密码明文存储（必须用 bcrypt 加密）

## Phase 4 — 核心链路联调

**目标：不依赖界面，把最重要的流程跑通**

### 做什么
用 Postman 或 curl 模拟完整用户流程：

```bash
# 1. 注册
POST /api/auth/register
{ "email": "test@test.com", "password": "123456" }
→ 返回 { success: true }

# 2. 登录
POST /api/auth/login
{ "email": "test@test.com", "password": "123456" }
→ 返回 { token: "eyJhb..." }

# 3. 激活
POST /api/activate
Header: Authorization: Bearer eyJhb...
{ "code": "XXXX-XXXX-XXXX" }
→ 返回 { product_key: "pro", expires_at: "2026-07-12" }

# 4. 鉴权检查
GET /api/auth/check
Header: Authorization: Bearer eyJhb...
→ 返回 { valid: true, permissions: [...] }
```

### 完成标志
- 上面4步全部跑通
- **这时候系统已经可以用了**（插件直接接入这几个接口）

## Phase 5 — 前端实现

**目标：给用户一个可操作的界面**

### 原则
```
界面是"皮"，接口是"骨"
骨架稳了，皮随时可以换
```

### 开发顺序
1. 先做**登录/注册**（其他页面都依赖登录态）
2. 再做**核心业务页面**（激活、查看状态）
3. 最后做**辅助页面**（设置、历史记录）

### 前端开发要点
```
状态管理：登录 token 存在哪里（localStorage / Cookie）
请求封装：统一处理 401（token 过期）、网络错误
路由守卫：未登录自动跳转到登录页
```

### 完成标志
- 用户通过界面能走完完整流程（注册→激活→使用）
- 不需要开发者介入

## Phase 6 — 测试上线

**目标：让真实用户能访问**

### 上线前检查清单
```
安全：
  □ 密码是否加密存储（bcrypt）
  □ SQL 注入防护（使用参数化查询）
  □ 敏感接口是否有限流（防暴力破解）
  □ JWT secret 是否换成强密钥
  □ HTTPS 是否开启

功能：
  □ 完整流程测试通过
  □ 错误情况有友好提示
  □ 手机端显示是否正常（如果需要）

运维：
  □ 数据库有备份策略
  □ 日志有记录
  □ 监控有告警（服务挂了能收到通知）
```

### 部署顺序
```
1. 服务器安装 Node.js、MySQL、Nginx
2. 上传代码，安装依赖
3. 配置环境变量（数据库密码、JWT secret 等）
4. PM2 启动后端服务
5. Nginx 配置反向代理 + SSL
6. 测试线上环境
```

## Phase 7 — 迭代扩展

**目标：根据实际使用反馈加功能**

### 原则
```
不要在没有用户的时候过度设计
先跑通核心，有用户了再根据反馈加功能
```

### 常见扩展路径
```
用户量小时：
  → 核心功能稳定，简单的管理后台
  
用户量增长：
  → 加渠道商体系
  → 加更多套餐差异化
  → 加数据统计

用户量大时：
  → 数据库优化（索引、读写分离）
  → 加缓存（Redis）
  → 考虑水平扩展
```

## 最容易犯的错误（总结）

```
1. 跳过 Phase 0/1，直接写代码
   → 写到一半发现方向不对，全部返工

2. 数据库表设计不稳定，边写边改
   → 牵一发动全身，越改越乱

3. 想把所有功能一次做完
   → 永远做不完，先做核心链路

4. 前端先行，后端跟不上
   → 前端 mock 了一堆假数据，接口对接时发现结构对不上

5. 不写接口文档，靠记忆
   → 过两周自己都忘了，维护成本极高
```

## 参考开源项目

构建 License 管理 SaaS 时，可以参考以下开源项目的设计思路：

### Lukittu ⭐ 85（GitHub: KasperiP/lukittu）

> 对标项目 — [github.com/KasperiP/lukittu](https://github.com/KasperiP/lukittu) | 线上服务：[lukittu.com](https://lukittu.com)

**定位：** 开源软件授权管理 SaaS（"锁定"，芬兰语）

```
和我们系统的对比：
  Lukittu:  Products → Licenses → Customers（通用，服务任意软件）
  本项目:   套餐 → 卡密/渠道码 → 用户（专用于 XHS Monitor 插件）
```

**技术栈：**
- Next.js App Router（API + Portal 合一，只需一个项目）
- PostgreSQL + Redis + Prisma ORM
- pnpm monorepo（apps/web + apps/bot + packages/shared）
- TypeScript 全栈（98.4%）
- Discord Bot（客服 + 通知集成）

**值得借鉴的设计：**
